Unity iPhone Basics

This section covers the most common and important questions that come up when starting to work with the iPhone.

Prerequisites

I've just received iPhone Developer approval from Apple, but I've never developed for the iPhone before. What do I do first?

A: Download the SDK, get up and running on the Apple developer site, set up your team, devices, and provisioning. We've provided a basic list of steps to get you started.

Should Unity-built games run in the iPhone Simulator?

A: No. The iPhone simulator is not very useful for Unity because it does not simulate all inputs from the iPhone and because it does not emulate performance you get on the iPhone. You test out gameplay directly inside Unity using the iPhone as a remote control while it's running the Unity Remote application. Then when you're ready to test performance and optimize the game, you publish to the iPhone.

Unity Features

How do I work with the touch screen and accelerometer?

A: In the scripting reference inside your Unity iPhone installation, you will find some classes called iPhoneInput, along with iPhoneSettings and a few others. These new iPhone-specific classes provide the hooks into the device's functionality that you need to build your games. Consult page on iPhone Input System for more information.

Does the iPhone support Unity's Terrain Engine?

A: Terrains are not currently supported in Unity iPhone. You need extremely low poly rendering for terrains on the iPhone. The terrain system is not made for that and you can do much better by doing extremely low poly terrains.

My existing particle systems seem to run really slowly on the iPhone. What should I do?

A: iPhone has relatively low fillrate. If your particles cover rather large portion of the screen with multiple layers - it will kill iPhone performance even with the simplest shader. We suggest baking your particle effects into a series of textures off-line. Then at run-time you can use 1-2 particles to display them via animated textures. You can get pretty good looking effects with minimum of overdraw this way.

Can I make a game that uses heavy physics?

A: Physics are pretty expensive on the iPhone, and we recommend you use them as little as possible. Physics does a lot of floating point number crunching which is not ideal for the iPhone's ARM processor. Completely avoid MeshColliders if at all possible, but they can be used if they are necessary. To improve performance, use a low Fixed framerate using Edit->Time->Fixed Delta Time. A framerate of 10-30 is recommended. Enable rigidbody interpolation to achieve smooth motion while using low physics frame rates. In order to achieve completely fluid framerate without oscillations, it is best to pick fixed deltaTime value based on the average framerate your game is getting on the iPhone. Either 1:1 or half the frame rate is recommended. For example if you get 30 fps, you should use 15 or 30 fps for fixed frame rate (0.033 or 0.066)

Can I access the gallery, music library or the native iPod player in Unity iPhone?

A: Yes - if you code it. Unity iPhone with the Advanced license supports the native plugin system, where you can add any feature you need, including access to Gallery, Music library, iPod Player and any other feature that the iPhone SDK exposes. Unity iPhone does not provide an API for accessing the listed features' through Unity scripts.

UnityGUI Considerations

What kind of performance impact will UnityGUI make on my games?

A: UnityGUI is fairly expensive when many controls are used. It is ideal to limit your use of UnityGUI to game menus, or very minimal GUI Controls while your game is running. It's important to note that every object that has a script with an OnGUI() call will require additional processor time, even if it is an empty OnGUI() block. It is best to disable any scripts that have an OnGUI() call if the GUI Controls are not being used. You can do this by marking the script as enabled = false.

Any other tips for using UnityGUI?

Try using GUILayout as little as possible. If you are not using GUILayout at all from one OnGUI call, you can disable all GUILayout rendering using MonoBehaviour.useGUILayout = false; This doubles GUI rendering performance. Finally, use as few GUI elements while rendering 3D scenes as possible.

Device Graphical Capabilities

Can I use bumpmapped objects in Unity iPhone?

A: iPhone is somewhere between TNT2 and the original GeForce in fragment processing capabilities. It can add or multiply two textures together, but normal mapping is simply not supported.

What are some basic vertex count guidelines?

A: To achieve 30 fps on the iPhone, you should keep per frame visible vertex count below 7000. That does not limit how many vertices you should have per scene. As long as you can ensure that you never have more than 7000 vertices visible at one time you, can guarantee a consistent high framerate. Take advantage of the new occlusion culling feature to ensure you stay in the 7000 vertices target.

General Performance

How I can improve application startup time on iPhone?

Application startup time one device is determined by your application binary size and first scene size. Application binary size can be improved by minimizing script dependencies and enabling stripping (Advanced license feature). More details on how to optimize binary size can be found here. User experience sometimes could be improved by adding small first scene that just shows some logo and then starts loading real scene. Scene size could be reduces by optimizing assets.

When running on iPhone my game framerate is bad. How I can improve it?

Your game framerate could be affected by many causes. It could be too complex geometry, use of OnGUI elements during gameplay, sub optimal scripts, too many .NET object allocations, etc. First step towards optimization is to identify your performance bottlenecks. Internal profiler is a tool that can help you. Next step is to make actual optimizations, you can read more about it here.

Xcode

I am getting a lot of compiler errors in Xcode. What's going on?

A: Make sure that you have selected the "Device" mode and not "Simulator" from Project->Set Active SDK in the menubar.

Sometimes when I click "Build and Go", I see strange errors.

A: This is a bug in Xcode, and is non-destructive to your project. Simply click OK to bypass the message, and your game should function completely normally.
